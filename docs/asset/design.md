## Problem Statement

在使用 Claude Code 时，会沉淀大量好用的技能包（`.claude/commands/*.md` 和 `.claude/skills/*/`）。这些技能散落在每个项目的 `.claude/` 目录下，跨项目管理极其痛苦：

- 在项目 A 里改了一个 command，需要手动 copy 到项目 B、C、D
- 更新后很难及时同步给其他项目
- 试过 symlink 但有局限（git 不跟踪内容、跨机器不可移植、无版本概念）
- 多个项目都改了同一个技能时，没有冲突检测机制

核心痛点是**跨项目同步更新**，不是技能发现或技能市场。

## What Makes This Cool

一个中心化的 AI Coding 技能管理平台，用「git 仓库做元技能数据源 + 轻量后端做协调」的混合架构，同时提供 Web 管理界面和 CLI 本地同步工具。

改一次，同步所有。再也不手动 copy。

## Glossary

- **元技能（Meta Skill）** — 存储在 git 技能仓库中的技能模板，是 source of truth
- **技能实例（Skill Instance）** — 同步到项目 `.claude/` 下的副本，由 `astack sync` 生成
- **技能仓库（Skill Repo）** — 存放元技能的 git 仓库，后端通过注册仓库地址来读取
- **目标项目（Target Project）** — 使用技能的项目，其 `.claude/` 目录由 CLI 管理
- **主工具目录（Primary Tool Dir）** — `.claude/`，作为项目中所有 AI 工具技能的 source of truth
- **衍生工具目录（Derived Tool Dir）** — `.cursor/`、`.codebuddy/` 等，通过 symlink 指向主工具目录

## Constraints

1. **先解决自己的痛点**，在自己满意的前提下再贡献给社区
2. **Web 和 CLI 同等优先级**，必须同时交付
3. **管理对象是纯文本文件**（markdown + 少量脚本），不是二进制产物
4. **单用户优先**，暂不考虑多租户、权限管理。后端绑定 `127.0.0.1`，不对外暴露
5. **Git 仓库是元技能的 source of truth**，后端服务不存储技能文件内容本身
6. **多 AI 工具兼容**：`.claude/` 是主目录，`.cursor/`、`.codebuddy/` 等通过 symlink 指向 `.claude/`，一份技能所有工具共用

## Premises

1. 核心痛点是"跨项目同步"，不是"技能发现"
2. 管理对象是 `.claude/commands/*.md` 和 `.claude/skills/*/` 两种结构
3. Git 仓库是元技能数据源，与后端服务分离；后端通过注册 git 仓库地址来获取技能内容
4. 用户现阶段就是自己
5. Web 和 CLI 必须同时做，同等优先级
6. 冲突是真实问题但不是高频问题，检测需要有，复杂自动合并可以延后

## Approaches Considered

### Approach A: Git-Native 方案（最小 viable）

中心仓库就是一个 git repo，后端是轻量 Node.js 服务包装 git 操作并暴露 REST API。Web 是 React SPA 调 API。CLI 用 git pull/push + file copy 做同步。

- Effort: M（2-3 周）
- Risk: Low
- Pros: Git 天然版本管理；数据存储免费；社区贡献自然
- Cons: 冲突合并依赖 git merge；Web 操作 git 需额外封装；扩展性受限

### Approach B: 独立后端 + 数据库方案（理想架构）

后端 + PostgreSQL，技能文件内容存数据库或对象存储。Web 完整管理。CLI 通过 API 拉取/推送。

- Effort: L（4-6 周）
- Risk: Medium
- Pros: 数据模型灵活；Web 功能强大；可自定义冲突逻辑；天然支持多用户
- Cons: 工程量大；技能文件脱离 git 生态；需维护后端运行时

### Approach C: 混合方案 — Git 仓库做存储，轻量后端做协调（选中）

Git 仓库是元技能 source of truth。后端服务只做协调层：项目注册（SQLite）、技能订阅关系、同步状态追踪、冲突检测通知。Web 读 git 仓库内容做展示，通过 API 管理配置。CLI 从 git 仓库拉取技能，写回时通过 API 上报变更。

- Effort: M-L（3-4 周）
- Risk: Low-Medium
- Pros: 技能文件留在 git 里；后端只管配置和协调；Web 可做丰富管理；CLI 逻辑简单
- Cons: 两套存储需保持一致性；需要 webhook 或轮询感知仓库变更

## Recommended Approach

**Approach C: 混合方案**

选择理由：各取所长。Git 天然擅长文本文件的版本管理、diff、merge，技能文件留在 git 里是最自然的选择。后端服务补上 git 做不好的事情 -- 项目注册、技能订阅关系（哪个项目用了哪些技能）、同步状态追踪（哪个项目的哪个技能是最新的）、冲突检测通知。

### 多 AI 工具兼容设计

不同 AI Coding 工具使用不同的配置目录：

| 工具 | 目录 | 结构 |
|------|------|------|
| Claude Code | `.claude/commands/`, `.claude/skills/` | 主目录（source of truth） |
| Cursor | `.cursor/commands/`, `.cursor/skills/` | symlink → `.claude/` |
| CodeBuddy | `.codebuddy/commands/`, `.codebuddy/skills/` | symlink → `.claude/` |

**策略：`.claude/` 是主工具目录，其他工具目录通过 symlink 指向 `.claude/`。**

```
my-project/
├── .claude/              ← 主目录（真实文件）
│   ├── commands/
│   │   ├── code_review.md
│   │   └── spec.md
│   ├── skills/
│   │   └── office-hours/
│   └── .astack.json
├── .cursor/              ← symlink
│   ├── commands → ../.claude/commands
│   └── skills → ../.claude/skills
└── .codebuddy/           ← symlink
    ├── commands → ../.claude/commands
    └── skills → ../.claude/skills
```

**Symlink 粒度：** 链接到 `commands/` 和 `skills/` 子目录级别，而不是整个 `.claude/` 目录。原因：各工具可能在自己的根目录下有其他配置文件（如 `.cursor/settings.json`），不应被覆盖。

**CLI 命令：**

```bash
# 初始化时自动创建 symlink（根据配置）
astack init
# → 创建 .claude/ 主目录
# → 根据 .astack.json 中配置的 tools 列表，自动创建 symlink

# 手动管理 symlink
astack link add cursor        # 创建 .cursor/ → .claude/ 的 symlink
astack link add codebuddy     # 创建 .codebuddy/ → .claude/ 的 symlink
astack link remove cursor     # 移除 .cursor/ 下的 symlink
astack link list              # 查看当前项目的所有 symlink 状态
```

**Web Dashboard 集成：** 项目管理页面中，每个项目显示已链接的工具列表，可以在 Web 上配置哪些工具需要 symlink。

### 数据模型（SQLite）

```sql
-- 注册的 Git 技能仓库（数据源）
CREATE TABLE skill_repos (
  id          INTEGER PRIMARY KEY,
  name        TEXT NOT NULL,
  git_url     TEXT NOT NULL UNIQUE,
  local_path  TEXT,              -- 后端 clone 到本地的路径
  last_synced DATETIME,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 注册的目标项目
CREATE TABLE projects (
  id          INTEGER PRIMARY KEY,
  name        TEXT NOT NULL,
  path        TEXT NOT NULL UNIQUE,  -- 项目本地路径
  primary_tool TEXT DEFAULT '.claude', -- 主工具目录名
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 项目的多工具 symlink 配置
CREATE TABLE tool_links (
  id          INTEGER PRIMARY KEY,
  project_id  INTEGER REFERENCES projects(id),
  tool_name   TEXT NOT NULL,         -- e.g. 'cursor', 'codebuddy'
  dir_name    TEXT NOT NULL,         -- e.g. '.cursor', '.codebuddy'
  status      TEXT DEFAULT 'active', -- 'active' | 'broken' | 'removed'
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(project_id, tool_name)
);

-- 元技能（从 git 仓库扫描得到）
CREATE TABLE skills (
  id          INTEGER PRIMARY KEY,
  repo_id     INTEGER REFERENCES skill_repos(id),
  type        TEXT NOT NULL,         -- 'command' | 'skill'
  name        TEXT NOT NULL,         -- e.g. 'code_review', 'office-hours'
  path        TEXT NOT NULL,         -- 在仓库中的相对路径
  version     TEXT,                  -- git commit hash
  updated_at  DATETIME,
  UNIQUE(repo_id, type, name)
);

-- 项目订阅了哪些技能
CREATE TABLE subscriptions (
  id          INTEGER PRIMARY KEY,
  project_id  INTEGER REFERENCES projects(id),
  skill_id    INTEGER REFERENCES skills(id),
  pinned_version TEXT,              -- null = 跟踪最新, 非空 = 锁定版本
  UNIQUE(project_id, skill_id)
);

-- 同步记录
CREATE TABLE sync_logs (
  id          INTEGER PRIMARY KEY,
  project_id  INTEGER REFERENCES projects(id),
  skill_id    INTEGER REFERENCES skills(id),
  direction   TEXT NOT NULL,        -- 'pull' | 'push'
  from_version TEXT,
  to_version  TEXT,
  status      TEXT NOT NULL,        -- 'success' | 'conflict' | 'error'
  conflict_detail TEXT,             -- 冲突详情（如果有）
  synced_at   DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### CLI 核心工作流

**初始化项目：**
```bash
cd /path/to/my-project
astack init
# → 在后端注册这个项目
# → 生成 .claude/.astack.json（项目配置文件）
```

**`.claude/.astack.json` 配置文件结构：**
```json
{
  "project_id": 1,
  "server_url": "http://127.0.0.1:7432",
  "primary_tool": ".claude",
  "linked_tools": ["cursor", "codebuddy"],
  "subscriptions": [
    { "repo": "my-skills", "type": "command", "name": "code_review" },
    { "repo": "my-skills", "type": "skill", "name": "office-hours" }
  ],
  "last_synced": "2026-04-19T11:30:00Z"
}
```

**技能引用格式：**
- 单仓库时：`astack subscribe code_review`（省略仓库前缀）
- 多仓库时：`astack subscribe my-skills/code_review`（`repo_name/skill_name` 消歧）

**订阅技能：**
```bash
astack subscribe code_review spec_review office-hours
# → 在后端记录订阅关系
# → 从 git 仓库拉取这些技能到 .claude/commands/ 和 .claude/skills/
```

**同步（拉取最新）：**
```bash
astack sync
# → 检查订阅的技能是否有新版本
# → 有更新 → diff 展示 → 确认后覆盖
# → 有冲突 → 提示手动解决或选择版本
```

**推送本地变更：**
```bash
astack push code_review
# → 检测本地 .claude/commands/code_review.md 与主仓库的 diff
# → 在主仓库创建 commit（message: "update: code_review from <project-name>"）
# → git push 到主仓库 main 分支（单用户场景直推 main）
# → 后端记录版本更新
```

**查看状态：**
```bash
astack status
# code_review    v3 (latest) ✓
# spec_review    v2 (behind, v3 available) ↓
# office-hours   v5 (local changes) ↑
```

### 版本语义

- **内部存储：** `skills.version` 存 git commit hash（精确标识）
- **展示层：** `v1`, `v2`, `v3` 是递增计数器，基于 `sync_logs` 中该技能的成功同步次数计算
- **映射关系：** `sync_logs` 每条记录都有 `from_version` 和 `to_version`（commit hash），展示层从记录数推导版本号

### 变更感知策略

v1 采用 **lazy pull** 模式：CLI 执行 `astack sync` 时触发 `git pull`，不依赖 webhook 或轮询。后续版本可加 webhook 实现实时推送通知。

### 冲突处理策略

1. **检测时机：** `astack sync` 和 `astack push` 时自动检测
2. **检测方式：** 对比本地文件 hash 与主仓库版本 hash，如果两边都有变更就是冲突
3. **解决方式（v1）：**
   - 展示 diff，让用户选择：保留本地 / 使用远程 / 手动合并
   - `astack resolve <skill-name> --keep-local | --use-remote | --merge`
4. **未来增强：** 基于 git three-way merge 做自动合并

### Web Dashboard 核心页面

1. **仓库管理** — 注册/移除 git 技能仓库，查看仓库中的所有技能
2. **项目管理** — 注册项目，查看每个项目的订阅列表
3. **技能矩阵** — 项目 × 技能的交叉视图，一眼看到谁用了什么、版本是否最新
4. **同步状态** — 实时同步状态看板，哪些项目需要更新，哪些有冲突
5. **版本历史** — 每个技能的变更历史，diff 查看

### 技术栈建议

- **后端：** Node.js + TypeScript + Hono（轻量 web 框架）+ better-sqlite3
- **Web：** React + TypeScript + Vite + TailwindCSS
- **CLI：** Node.js + TypeScript + Commander.js
- **Git 交互：** simple-git（Node.js git wrapper）
- **Monorepo 管理：** 三个 package（server、web、cli）放在同一个 repo

### 后端部署与 CLI 协作

- 后端绑定 `127.0.0.1:7432`，本机运行，不对外暴露
- CLI 每次执行命令时自动检测后端是否在线（HTTP health check）
- 如果后端不在线，CLI 自动后台拉起 `astack server`（daemon 模式）
- `astack server` 同时 serve 后端 API 和 Web 静态资源（开发阶段打包在一起）
- SQLite 使用 WAL 模式 + busy timeout（5s），避免 Web 和 CLI 并发写入时的 `SQLITE_BUSY`

### Git 仓库交互优化

- 注册仓库时使用 `git clone --depth 1`（shallow clone）减少首次 clone 开销
- 后续 `astack sync` 时 `git pull` 只拉增量

### 错误处理

| 场景 | 行为 |
|------|------|
| 后端不可达 | CLI 自动拉起后端；如拉起失败，报错退出并提示 `astack server` |
| Git pull/push 失败 | 报错并展示 git 错误信息，不写入 sync_logs |
| 文件写入失败（权限） | 回滚已写入的文件，报错退出 |
| 目标项目 `.claude/` 不存在 | 自动创建目录结构 |
| 技能仓库不可达 | 使用本地缓存版本，标记为 stale，下次重试 |

## Design Review 决策（Web Dashboard UI/UX）

经 `/plan-design-review` 锁定。7 个 pass，初评 3/10 → 终评 9.6/10。

### 1. 信息架构

**主导航：左侧 fixed sidebar (240px)** + 主内容自适应宽度，max-width 1400px centered。

```
  ▾ ASTACK                 [server: 🟢 online]

  📦 Repos                  (N)
  📁 Projects               (N)
  ⚡ Sync Status            [N 项待关注]

  ────
  ⚙ Settings
  📘 Docs
```

**默认页 = `/` Sync Status**（工作台视图，打开立刻看到需要处理的事）。

**5 个页面的信息层级 Priority 1/2/3：**

| 页面 | Priority 1 | Priority 2 | Priority 3 |
|-----|-----------|-----------|-----------|
| Sync Status | 需关注的 project × skill（冲突 / 落后 / 未推送） | 最近同步记录 | 全局项目 × 技能视图（折叠） |
| Repos | 已注册仓库列表（HEAD hash + 最后拉取时间） | 仓库内 command / skill 数 | 注册新仓库操作 |
| Projects | 项目列表（路径 + 订阅数 + 同步状态） | Tool links 图标群 | 注册新项目操作 |
| Skill Matrix | 交叉表（项目行 × 技能列） | 按仓库 / 项目筛选 | 批量操作 |
| Version History | 选定技能的 git log | Diff viewer | 锁定到历史版本 |

**跨页面跳转链路：**
- Sync Status 某行 → Skill Matrix 对应单元格 → Version History
- Project 卡片 → Project 详情页（含 Linked Tools 区块）

## Engineering Review 决策

经 `/plan-eng-review` 锁定，11 个架构 / 代码质量 / 测试 / 性能决策。

### 架构决策

#### 1. Git 存储模型：显式双副本

技能文件存在两个位置，边界清晰：

```
        远程 git 仓库
             │
        (clone/pull/push)
             │
             ▼
~/.astack/repos/<repo-name>/     ← [upstream 镜像]
  commands/code_review.md         （后端持有，与远程 git 同步）
  skills/office-hours/SKILL.md
             │
        (copy/diff)
             │
             ▼
<project>/.claude/                ← [working copy]
  commands/code_review.md         （目标项目实际使用的副本）
  skills/office-hours/SKILL.md
```

**搬运规则：**
- `astack sync` — 远程 → upstream 镜像 → working copy
- `astack push` — working copy → upstream 镜像 → 远程
- working copy 不受目标项目的 git 追踪（加入目标项目的 `.gitignore`）
- upstream 镜像是后端独占，CLI 不直接操作

#### 2. Source of Truth 层级：文件/Git 优先

| 信息类型 | Source of Truth | SQLite 角色 |
|---------|----------------|------------|
| 技能文件内容 | 远程 git 仓库 | 不存储 |
| 技能当前版本 | upstream 镜像的 git HEAD | 缓存 `skills.version` |
| 项目订阅关系 | `<project>/.claude/.astack.json` | 镜像到 `subscriptions` 表 |
| 同步历史 | 只在 SQLite（`sync_logs`） | source |
| Symlink 状态 | 文件系统实际状态 | 缓存到 `tool_links` 表 |

**冲突处理：** SQLite 被动向文件/git 对齐。`astack sync --refresh` 清空 [CACHE] 表并从 git/文件重建。

#### 3. Symlink 跨平台：只支持原生 symlink

- macOS/Linux：默认直接用 symlink
- Windows：要求开启开发者模式（文档中明确说明）；不做 copy fallback
- symlink 粒度：`.cursor/commands → ../.claude/commands`、`.cursor/skills → ../.claude/skills`（子目录级别，避免破坏各工具自己的根配置）

#### 4. 后端 Daemon：常驻 daemon，手动启动

- 用户首次安装后需手动 `astack server start`（文档给出 launchd/systemd 可选配置示例作为附录）
- CLI 触发命令时检测 `127.0.0.1:7432` 连通性：不通 → 立即报错并提示 `astack server start`
- PID 文件：`~/.astack/daemon.pid`；日志：`~/.astack/daemon.log`；锁文件：`~/.astack/daemon.lock`
- 子命令：`astack server start | stop | status | logs`

#### 5. 并发写入锁：仓库级互斥锁

- 后端内存维护 `Map<repoId, Mutex>`
- 对同一 `repo_id` 的所有写操作（pull/push/文件写入）走队列
- 等待超时 30 秒，超时后返回错误 `REPO_BUSY`
- CLI 表现：`waiting for repo lock on my-skills...`；Web 表现：loading spinner

#### 6. 多仓库命名空间：全限定名 + 冲突拒绝

- 内部标识：`<repo-name>/<type>/<skill-name>`，如 `my-skills/command/code_review`
- CLI 使用：单仓库时允许短名 `astack subscribe code_review`；多仓库同名必须显式 `astack subscribe my-skills/code_review`
- 项目文件布局保持平铺（`.claude/commands/code_review.md`），与 AI 工具兼容
- 一个项目不能同时订阅两个同名技能（会报错要求用户重命名或选一个）

### 代码质量决策

#### 7. SQLite 表按 CACHE/SOURCE 分类

```sql
-- [CACHE] 可从 git/文件重建
CREATE TABLE skill_repos (...);     -- 仅 git_url 是 source，其他字段可重建
CREATE TABLE skills (...);           -- 全部可从 git 扫描重建

-- [SOURCE] 权威数据
CREATE TABLE projects (...);         -- 项目注册表，本地唯一
CREATE TABLE subscriptions (...);    -- 与 .astack.json 互为镜像（见下）
CREATE TABLE sync_logs (...);        -- 只在 SQLite
CREATE TABLE tool_links (...);       -- symlink 状态缓存
```

**重建策略：** `astack sync --refresh` 清空所有 [CACHE] 表，重新扫描已注册的 git 仓库重建；[SOURCE] 表保留。

**subscriptions 一致性：** `.astack.json` 和 `subscriptions` 表互为镜像。每次 `astack sync` 时先从 `.astack.json` 同步到 SQLite（文件优先），再基于 SQLite 执行业务逻辑。

#### 9. 技能仓库结构标准：默认镜像 `.claude/`，支持显式 layout 覆盖

**默认结构（`DEFAULT_SCAN_CONFIG`）：**

```
<skill-repo>/
├── commands/              ← 扫描规则：*.md 文件 = command 技能
│   ├── code_review.md
│   └── spec.md
├── skills/                ← 扫描规则：子目录（含 SKILL.md）= skill 技能
│   └── office-hours/
│       └── SKILL.md
└── astack.yaml            ← 可选，元数据清单
```

**扫描抽象（v0.2 起）：** 每个 `skill_repo` 持有一个可选的 `scan_config: ScanConfig | null`，为 null 时退化为默认结构。`ScanConfig` 形如：

```ts
{
  roots: Array<{
    path: string,                     // 相对 repo 根，"" 表示 repo 根本身
    kind: 'skill-dirs'                // 子目录含 SKILL.md → type='skill'
        | 'command-files'             // *.md → type='command'
        | 'agent-files'               // *.md → type='agent'
  }>
}
```

**标准布局预设（任何仓库注册时如果省略 scan_config 都走这个）：**
```ts
DEFAULT_SCAN_CONFIG = {
  roots: [
    { path: 'skills',   kind: 'skill-dirs' },
    { path: 'commands', kind: 'command-files' }
  ]
}
```

**另外支持的布局：**

- **Flat（扁平）**：`[{ path: '', kind: 'skill-dirs' }]` —— 每个根目录下的子目录（含 SKILL.md）就是一个 skill。用于 `garrytan/gstack` 这类仓库。
- **Multi-root（多根）**：`[{skills,skill-dirs}, {commands,command-files}, {agents,agent-files}]` —— 用于 `affaan-m/everything-claude-code` 这类含 `agents/` 的仓库。

**白名单原则：** 一个目录被识别为 skill 当且仅当含 `SKILL.md`。这让 `bin/`、`lib/`、`docs/`、`test/` 等非技能目录被自动过滤，无需维护黑名单。

**非递归：** flat 扫描只看根目录第一层，不深入。避免符号链接环和误扫 `test/` 下的 fixture。

**Frontmatter：** 每个 SKILL.md / command.md 的 YAML frontmatter 里 `name` 和 `description` 会被读取。目录/文件名为权威（冲突时目录名胜出，但产出 warning）。描述存入 `skills.description` 字段供 Dashboard 展示。

**扫描规则（概要）：**
- `skill-dirs` 根：子目录 + 含 SKILL.md = 一个 type='skill' 技能，目录名为技能名
- `command-files` 根：`*.md` = 一个 type='command' 技能，文件名去 `.md` 为技能名
- `agent-files` 根：`*.md` = 一个 type='agent' 技能，文件名去 `.md` 为技能名
- 可选的仓库根 `astack.yaml` 提供元数据（作者、描述、版本约束、依赖）

**可选清单 `astack.yaml` 示例：**
```yaml
name: my-skills
description: My personal AI coding skills
author: alexjhwen
commands:
  - name: code_review
    description: Code review checklist
skills:
  - name: office-hours
    description: YC-style product office hours
```

**兼容性：** 用户可以把现有项目的 `.claude/` 目录直接 `cd .claude && git init && git remote add ...` 变成技能仓库。

### 性能决策

#### 11. 缓存 TTL + SSE 推送

**Git pull 策略：**
- 后端缓存每个仓库的 HEAD hash，TTL 5 分钟
- `astack sync` 命中缓存时跳过 `git ls-remote` 探测
- `astack sync --force` 强制刷新所有缓存
- 多仓库 pull 并行（`Promise.all`），单仓库失败不影响其他

**Web 刷新策略：**
- Server-Sent Events（SSE）端点 `/api/events` 推送状态变化
- CLI 做写操作后，后端主动广播事件（`skill-updated` / `conflict-detected` / `sync-completed`）
- 无变更时不推，Web 零轮询

**SQLite 配置：**
- WAL 模式
- busy timeout 5 秒
- 读写语句分池（better-sqlite3 的 prepared statements）

## NOT in Scope

本次 v1 明确不做：

- **多用户 / 多租户 / 权限管理** — 单用户本机工具
- **服务器端云部署** — 后端只跑 localhost
- **技能市场 / 发现** — Astack 是同步工具，不是市场
- **自动 three-way merge** — v1 只做冲突检测 + 手动选择（keep-local / use-remote / 手动 merge）
- **Webhook 实时推送** — v1 lazy pull + TTL 缓存 + SSE 本地事件
- **Windows 上的 copy fallback** — v1 只支持 symlink，Windows 要求开发者模式
- **技能间依赖** — 不做 skill A 依赖 skill B 的声明式依赖管理
- **技能评分 / 评论** — 不是社区市场
- **订阅版本锁定的自动升级建议** — v1 用户自己决定何时 sync

## Failure Modes

每个新代码路径的一种真实生产失败场景：

| 代码路径 | 失败场景 | 测试覆盖 | 错误处理 | 用户可见性 |
|---------|---------|---------|---------|-----------|
| SyncService.push | git push 被远端拒绝（权限 / 冲突） | ✅ E2E | ✅ 返回 git stderr | ✅ 清晰错误 |
| SyncService.pull | 远端仓库不存在 / 网络失败 | ✅ UNIT | ✅ 使用缓存 + 标记 stale | ✅ CLI 警告 |
| SymlinkService.create | Windows 非管理员模式 | ✅ UNIT | ✅ 提示开发者模式 | ✅ CLI 引导 |
| LockManager.acquire | 死锁（前一个操作卡住） | ✅ UNIT | ✅ 30 秒超时 `REPO_BUSY` | ✅ CLI/Web 提示 |
| CLI → daemon | Daemon 进程崩溃但 PID 文件残留 | ✅ UNIT | ✅ 端口检测确认 | ✅ 提示 `astack server start` |
| RepoService.scan | 仓库中有非法文件名 / 空 SKILL.md | ✅ UNIT | ✅ 跳过并记录日志 | ⚠️ 仅日志（低严重度） |
| SQLite 写入 | 磁盘满 | ❌ 未覆盖 | ❌ 未处理 | ❌ **CRITICAL GAP** |

**Critical gap：** 磁盘满的场景。实现时需在写 SQLite / 文件前做磁盘空间检查，或捕获 `ENOSPC` 给出清晰错误。补充到 Implementation TODOs。
